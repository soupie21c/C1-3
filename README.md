# C1-3
GenAI 기초 3 (B1-3): 노코드 자동화 기초 — 워크플로우 설계

[프로젝트 1] 자동화 도구 비교 구현

동일 워크플로우 (Make vs n8n 2종)

워크플로우: "고객 피드백 Google Form 응답 → AI 감성 분류 → 부정적 피드백은 Slack 긴급 알림, 긍정적 피드백은 Gmail 자동 감사메일 → 모든 응답 Google Sheet 기록"


Trigger(1): Google Forms 새 응답 (폼 필드: 이름/이메일/피드백 내용/만족도 1~5점)
Action(3+): ① AI로 피드백 텍스트 감성 분류(긍정/부정) ② 분류 결과에 따라 Slack 알림 또는 Gmail 자동 발송 ③ Google Sheet에 원본 응답 + 분류 결과 행 추가
조건 분기(1+): 감성 분류 결과 = 부정 → #cs-urgent 채널 알림, 긍정 → 응답자에게 감사 이메일 자동 발송
각 분기 경로 1회+ 실행 확인 (스크린샷 make-run.png / n8n-run.png): 부정 피드백 1건 → #cs-urgent 알림 도착 확인, 긍정 피드백 1건 → 감사 이메일 발송 로그 확인


비교 분석 보고서 (비교 항목 7개)

비교 항목Maken8n (Cloud 무료 / 자가호스팅)① UI/UX시각적 모듈 캔버스, 색상 구분 명확노드형 캔버스, 코드 노드 삽입 가능해 다소 개발자 친화적② 설정 난이도중 (분기·매핑 설정이 직관적)중~상 (JS 표현식 문법 학습 필요하지만 유연)③ 연동 서비스 범위앱 커넥터 다수 + HTTP 모듈로 보완기본 노드 수는 적지만 HTTP/커스텀 코드로 사실상 무제한④ 무료 플랜 범위1,000 ops/월 (클라우드 고정)Cloud 무료 체험 제한적이나 자가호스팅 시 실행 횟수 무제한⑤ 실행 로그 확인 방식시나리오별 실행 히스토리, 모듈 단위 입출력 확인실행(Execution) 로그에서 노드별 입출력 JSON 상세 확인 가능⑥ 조건 분기/반복Router · Filter · Iterator 내장, 설정 쉬움IF/Switch 노드 + 코드 노드로 복잡한 조건도 세밀 제어⑦ 확장성(코드/커스텀)제한적 (주로 노코드 범위 내)Function 노드로 JS 코드 직접 삽입 가능, 코드 기반 확장에 유리

장단점


Make: 화면이 직관적이라 진입장벽이 낮고 팀 인수인계가 쉽지만, 무료 실행 횟수(월 1,000 ops)가 고정되어 있어 대량·고빈도 워크플로우에는 부담. 완전한 코드 삽입은 불가.
n8n: 자가호스팅 시 실행 횟수 제약이 사실상 없고 코드 노드로 세밀한 로직 구현이 가능하지만, 서버 관리 부담이 있고(자가호스팅 기준) 노드 문법(표현식)에 익숙해지는 데 시간이 걸림.


적합 상황


비개발자 중심 팀, 빠른 시각적 파악과 협업이 중요한 경우 → Make
실행 빈도가 높거나 향후 코드 기반 로직 확장이 예상되는 경우, 서버 운영 리소스가 있는 경우 → n8n



[프로젝트 2] 자유 주제 자동화 설계·구현

반복 업무 정의: 온라인 스토어에서 신규 주문이 들어올 때마다 담당자가 수동으로 주문 금액을 확인해 VIP 주문은 포장팀에 별도로 알리고, 일반 주문은 주문 관리 시트에 옮겨 적는 작업 (1건당 약 3분, 하루 평균 20건 → 1시간 소요).

선정 도구: n8n — 선정 이유: 주문 건수가 많아질수록 무료 실행 한도에 걸리는 Make보다, 자가호스팅으로 실행 횟수 제약이 없는 n8n이 장기적으로 유리하고, 금액 구간이 늘어나도 Function 노드로 분기 로직을 코드로 유연하게 확장 가능.

워크플로우 설계 (흐름)

[Trigger] 신규 주문 Webhook 수신 (쇼핑몰 주문 시스템 → n8n Webhook URL)
   → [Set] 주문 데이터 정리 (주문번호/금액/상품/주소)
   → [IF] 주문 금액 10만원 이상?              ← 조건 분기
       ├─ 예(VIP) → [Slack] #vip-packing 채널 알림 (우선 포장 요청)
       │              → [Sheet] "VIP주문" 시트에 행 추가
       └─ 아니오(일반) → [Sheet] "일반주문" 시트에 행 추가
   → (공통) [Slack] #order-log 채널에 처리 완료 로그 전송

구현/실행 (스크린샷 p2-config.png / p2-run.png): Webhook 트리거 발화 확인 → 15만원 주문 1건 → VIP 분기 → #vip-packing 알림 + VIP 시트 기록 확인 / 3만원 주문 1건 → 일반 분기 → 일반 시트 기록 확인.


평가기준 대응 (설명형)

(기준 7) 두 도구 동일성 설계 기준

Make와 n8n의 기능 구현 방식 차이를 극복하고 동일성(입력 이벤트/핵심 Action/분기/최종 출력)을 맞추기 위해 적용한 기준:


입력 이벤트 고정: 두 도구 모두 동일한 Google Form을 Trigger로 사용(폴링 주기 차이는 허용).
핵심 Action 동치: 감성분류(AI) → 알림 → 시트기록 3액션을 동일 순서로 배치. Make는 모듈, n8n은 노드로 1:1 매핑.
분기 동치: Make=Router, n8n=IF 노드로 동일 조건(감성 긍/부정)·동일 2경로 구성.
출력 동치: 두 도구 모두 Slack 알림 문구 포맷 + Sheet 컬럼 스키마를 동일하게 통일.


(기준 8) P2 워크플로우 단계별 입출력

단계입력출력Trigger(Webhook)신규 주문 payload실행 트리거Set원본 주문 JSON정리된 필드(금액/상품/주소)IF(분기)주문 금액VIP/일반 경로Slack/Sheet정리된 주문 정보알림 메시지 + 시트 행

(기준 9) 조건 분기 설계 기준


P1의 Router: 감성 분류 결과(긍정/부정) = 응답 대응 방식 분기 기준. 부정 피드백은 신속 대응이 필요하므로 즉시 알림, 긍정은 자동 감사 인사로 처리.
P2의 IF: 주문 금액 10만원 기준 = 포장 우선순위 분기 기준. VIP 주문은 즉시 알림이 필요하지만 일반 주문은 배치성 기록으로 충분.


(기준 10) Trigger vs Action 차이 (예시)


Trigger = 시작점: 외부 상태 변화를 감지해 워크플로우를 깨움. 예) "신규 주문 도착". 스스로는 아무 처리도 하지 않음.
Action = 처리: Trigger 이후 실제 작업 수행. 예) "감성 분류", "Slack 전송". Trigger 없이는 실행되지 않음.
비유: Trigger = 매장 초인종(손님 방문 감지), Action = 응대 직원이 안내·계산하는 후속 처리.


(기준 11) 도구 차이를 업무 요구사항 관점에서 연결


협업: 비개발 팀원과 함께 관리해야 하는 워크플로우는 Make의 시각적 캔버스가 유리.
디버깅: n8n은 노드별 JSON 입출력을 그대로 볼 수 있어 로직 오류 추적이 빠름.
확장: 분기 조건이 코드 수준으로 복잡해질 것으로 예상되면 n8n의 Function 노드가 유리.
비용: 실행 빈도가 낮고 단순한 경우 Make 무료 한도로 충분, 고빈도·대량 실행은 n8n 자가호스팅이 경제적.


(기준 12) P2 도구 선택 이유 (비교 항목 2개+ 연결)


항목 ④ 무료 플랜 범위: 하루 20건 이상 주문이 지속 발생 → Make의 월 1,000 ops 한도보다 n8n 자가호스팅의 무제한 실행이 부합.
항목 ⑦ 확장성(코드/커스텀): 향후 VIP 기준이 금액 외에 "구매 이력" 등으로 복잡해질 가능성이 있어, Function 노드로 로직을 코드화할 수 있는 n8n이 유리.


(기준 13) 트리거 불안정/외부 연동 실패 시 모니터링·재시도·대체 경로


모니터링: n8n 워크플로우에 Error Trigger 워크플로우를 별도 연결해 실패 시 #ops-alert Slack 채널로 어느 노드에서 어떤 에러가 났는지 전송.
재시도: 노드별 자동 재시도 설정(최대 3회, 지수 백오프), 초과 시 실패 항목을 별도 "실패 큐" 시트에 적재해 수동 재처리.
대체 경로: Webhook 트리거 유실 대비, 5분 주기 폴링 백업 워크플로우로 누락 주문 재확인. Sheet 저장 실패 시 임시 로컬 CSV(또는 백업 시트)에 우선 적재 후 복구 배치로 재이관.
중복 방지: 주문번호를 Data Store(또는 별도 시트)에 기록해 재시도 시 동일 주문이 중복 적재되지 않도록 체크.


(기준 14) 분기 증가/복잡화 시 구조화·모듈화


서브워크플로우 분리: "주문 수신", "분기 판단", "알림/기록"을 별도 n8n 워크플로우로 쪼개고 Webhook으로 연결해 하나의 워크플로우가 비대해지지 않도록 함.
IF → 룩업 테이블 전환: VIP 기준이 금액 1단계에서 등급별(브론즈/실버/골드) 다단계로 늘어나면, 다중 IF 대신 "등급 → 처리방식" 매핑을 시트/DB로 관리해 데이터 주도 방식으로 전환.
공통 모듈 재사용: 알림 포맷팅, 중복 체크 로직 등은 공용 서브워크플로우로 분리해 재사용.


(기준 15) 노코드 한계 업무 + 코딩 기반 확장 아이디어


한계 업무: "수천 건의 과거 피드백 텍스트를 임베딩 기반으로 유사 이슈끼리 클러스터링해 반복 불만 유형을 자동 도출"하는 작업. 노코드 도구는 대량 벡터 연산, 배치 처리, 상태 관리에 취약하고 실행 시간/ops 한도에 걸림.
코딩 확장 아이디어: Python 배치(cron) + 벡터DB(pgvector 등)로 임베딩·유사도 클러스터링을 수행하고, 결과 요약만 n8n/Make를 통해 Slack·Sheet로 전달. 즉 "트리거·단순 분기·알림"은 노코드가, "대량·연산집약적 분석"은 코드 기반이 담당하는 역할 분담.



보너스 (수행)


AI 연동 Action(요약/분류): P1은 피드백 텍스트를 AI로 긍정/부정 감성 분류하는 Action을 삽입(make-run.png, n8n-run.png에서 분류 결과 확인). P2는 향후 확장 시 주문 상품 카테고리 자동 분류에도 동일 AI Action을 재사용할 수 있도록 설계.
실패 알림 + 재시도 전략(기준 13 상세): Error Trigger 워크플로우 → #ops-alert Slack 알림, 노드별 자동 재시도 3회(지수 백오프), 초과 시 실패 큐 시트 보류, Webhook 유실 대비 폴링 백업, 주문번호 기반 중복 적재 방지.



동작 구조 원리 (요약)


Trigger-Action 모델: Trigger(webhook/polling) → 조건 분기 → Action 체인. 시각적 노드로 코드 없이(또는 최소 코드로) 이벤트 파이프라인을 구성.
안정성: 중복 방지 키, 재시도 상한, 별도 에러 알림 경로로 무인 운영 시에도 실패를 놓치지 않도록 설계.


제약/환경 기록
